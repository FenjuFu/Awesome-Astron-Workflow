# 用户案例：某电信运营商地市分公司 Astron RPA 内网验证码识别

*信息来源：项目沟通记录及用户反馈 (2026-01-13 ~ 2026-02-05)*
---

## 1. 案例概况 (Case Overview)
*   **项目名称**：
*   **客户方**：某电信运营商地市分公司
*   **应用产品**：Astron RPA, Astron Agent
*   **部署状态**：生产环境（灰度/试运行）
*   **核心场景**：自动业务受理、业务稽核

## 2. 业务背景 (Background)
某电信运营商地市分公司致力于通过自动化技术提升业务处理效率，目前正重点推进**自动业务受理**与**业务稽核**场景的落地。由于电信行业对数据安全的高标准要求，所有开发与运行环境均部署于企业**内网**，与互联网物理隔离，这对开发工具的灵活性和扩展性提出了极高要求。

## 3. 挑战与痛点 (Challenges)
在推进 RPA 流程开发的过程中，技术团队遇到了以下关键瓶颈：
*   **内网环境依赖缺失与版本兼容性**：业务流程中需要集成第三方 Python 库（如 **`ddddocr`**）来实现验证码识别（OCR）功能。由于内网无法使用 `pip` 在线安装，且 Astron 运行环境（Python 3.13.5）与部分第三方库存在版本兼容性问题（如 `ddddocr` 对 Python 版本有特定要求），导致常规的离线安装包（`.whl`）无法直接安装或运行。
*   **手动集成失效**：开发人员尝试将库文件直接复制到项目 `lib` 目录下，但未能被 Astron RPA 项目的隔离虚拟环境正确识别，导致代码无法运行，阻碍了流程的调试与上线。
*   **复杂元素识别困难**：在处理动态业务系统页面时，用户需要基于特定的文本内容（如中文名称）定位数据行。然而，RPA 工具的元素探测器默认未展示所有属性节点，导致用户难以通过属性筛选（如 `innerText`）精确定位目标元素，增加了开发难度。
*   **版本升级变更影响**：产品版本更新后，部分底层逻辑发生变更（如 Python 脚本的返回值机制由 `return` 改为配置参数输出），但缺乏显著的变更说明，导致存量流程运行异常。
*   **动态 Iframe 识别局限**：在处理嵌套 Iframe 的动态页面时，Iframe 的 XPath 路径存在随机变化（如 `position()` 索引变动），当前元素探测器无法生成兼容性强的通用选择器，用户需编写复杂的容错逻辑（Try-Catch）来应对。

## 4. 解决方案 (Resolution)
针对内网隔离环境的特殊性及开发痛点，Astron 技术支持团队提供了全方位的技术支持与方案：
1.  **底层依赖注入方案**：
    *   **底层注入**：将兼容的第三方库直接集成安装至 RPA 引擎底层的 **`python_core`** 基础环境中。
    *   **环境重置与重建**：指导用户清理工程目录下的 **`venv`** 文件夹并重启工程，触发 Astron 引擎基于更新后的 `python_core` 自动重建虚拟环境，确保第三方库在项目中生效。
2.  **全流程技术辅导**：针对元素识别难题，支持团队协助用户分析页面 DOM 结构，通过父级元素定位等方式解决动态元素的抓取问题；指导用户适配新版引擎的参数配置模式，解决 Python 扩展脚本的数据传递问题。
3.  **运维规范建议**：针对内网 Docker 环境的升级风险，协助用户梳理了容器备份与回滚的标准操作流程，保障生产环境变更安全。

## 5. 客户价值 (Value)
*   **突破内网开发限制**：建立了一套在企业内网隔离环境下灵活扩展 Python 第三方能力的标准化流程，并解决了 Python 版本兼容性带来的安装难题。
*   **保障核心业务落地**：成功解决了验证码识别等技术卡点，确保了“自动业务受理”与“业务稽核”等核心自动化流程的顺利灰度上线与稳定运行。

## 6. 产品共创与反馈 (Co-creation & Feedback)
在项目支持过程中，客户基于实际业务场景提出了宝贵的产品改进建议，为 Astron 产品的持续迭代提供了方向：
*   **依赖分发机制**：建议增加服务器端统一安装第三方库并分发至客户端的功能，以降低团队内网环境下的配置成本。
*   **元素探测优化**：建议优化元素探测器，默认展示更全的属性信息（如文本内容等），以便用户更直观地配置选择器。
*   **组件共享能力**：提出了团队内部自定义组件（如 OCR 模块）的共享需求，期望能通过团队库的方式实现能力的复用。
*   **AI 能力集成需求**：用户希望能接入企业内网自建的大模型（如千问）API，并期望利用视觉/AI 技术实现“所见即所得”的无代码元素控制。
*   **变更管理优化**：强烈建议在发布新版本时提供详细的“非兼容性变更说明”，避免因底层逻辑调整导致业务中断。
*   **智能选择器增强**：建议增强对动态 Iframe 及复杂结构页面的智能识别能力，提升选择器的鲁棒性。
*   **项目迁移与备份**：鉴于服务器迁移场景（如操作系统变更），用户建议增加“导出项目到本地”及“从本地导入项目”的功能，以便于代码备份和跨环境迁移，目前需依赖数据库层面的数据迁移。
*   **触发器稳定性修复**：针对邮件触发器在特定逻辑组合（如 OR 条件）下的误判问题（如错误识别附件状态），建议尽快排查底层逻辑bug，确保触发的精准度。
*   **C/S 架构应用支持**：用户新增了对数据库客户端（如 PL/SQL, DBeaver 等 C/S 架构软件）的自动化需求，期望产品能提供针对此类非 Web 应用的更明确的原子能力支持或最佳实践指导。
*   **跨环境分发兼容性优化**：建议优化应用市场的分发与环境初始化机制，确保应用在不同账号和全新服务器环境获取后，浏览器插件和元素捕获能力的稳定一致。
*   **元素操作精准度提升**：建议排查并修复在特定页面结构下点击动作出现的坐标偏移问题，确保动作执行位置与探测锁定位置的绝对一致。
