# 用户案例：某通信设备制造企业 Astron RPA 离线内网自动化

*信息来源：项目沟通记录 (2026-03-04 ~ 2026-04-02)*

---

## 1. 案例概况 (Case Overview)
*   **项目名称**：某通信设备制造企业 RPA 自动化项目
*   **客户方**：某通信设备制造企业
*   **应用产品**：Astron RPA
*   **部署状态**：POC / 试运行
*   **核心场景**：离线环境自动化、界面录制与回放、AI能力集成

## 2. 业务背景 (Background)
某通信设备制造企业正在探索使用 Astron RPA 进行业务流程自动化。由于业务涉及涉密信息，运行环境为严格的**离线内网环境**，且客户端部署在配置受限的涉密终端上。这要求 RPA 平台必须具备完整的离线部署能力，并且能够在特定的硬件环境下稳定运行。此外，客户期望引入内部部署的 AI 大模型来进一步提升 UI 自动化开发效率，并与内部 Jenkins 等 CI/CD 工具打通。

## 3. 挑战与痛点 (Challenges)
在项目推进过程中，技术团队遇到了以下关键问题：
*   **离线环境部署难题**：由于内外网物理隔离，用户在将 Docker 镜像导入离线环境时遇到阻碍，初期导入的镜像出现无名称或无法识别的情况，导致服务无法正常启动。
*   **涉密终端性能瓶颈**：在涉密客户端（16G内存）上运行 RPA 客户端软件时，出现了严重的卡顿现象，甚至影响了操作系统的正常响应，而在普通个人电脑上则运行流畅。
*   **复杂交互元素识别**：
    *   **输入框事件触发失效**：在处理网页登录界面时，RPA 填入账号密码后，页面未检测到输入行为，“登录”按钮仍保持置灰（不可点击）状态。
    *   **HTTPS 安全拦截**：内网访问特定 HTTPS 页面时弹出浏览器安全警告，阻断了自动化流程。
    *   **元素定位困难**：部分页面元素无法通过常规选择器捕获，且 XPath 在 RPA 中定位不唯一，导致运行不稳定。
*   **智能录制使用困惑**：用户在使用“智能录制”功能后，未能直观找到生成的流程文件，且发现录制结果中部分元素未能被有效识别（显示为 `--`）。
*   **变量类型与参数传递机制不清晰**：用户在尝试将主流程变量传递给自定义组件时，不清楚如何在组件内定义入参以及在主流程中进行赋值绑定。此外，在处理脚本返回值、时间相关动态变量赋值以及变量类型判断（如整数 `0` 的相等判断需强制转换）时遇到障碍。
*   **AI 能力集成受限**：用户希望在 RPA 中集成内网部署的自定义 AI 模型以提升开发效率，但初期遇到配置入口隐蔽及服务端支持暂不完善的问题。
*   **团队多用户协作问题**：在多用户协作场景下，由于底层数据库架构调整（Casdoor 多数据源配置问题），导致市场邀请和用户查询功能出现异常，团队成员邀请失败。
*   **底层组件稳定性问题**：在执行“关闭浏览器”等基础组件时，用户环境稳定触发底层 DLL 调用失败报错。
*   **动态元素定位与 XPath 维护疑虑**：用户对元素定位的底层逻辑存在疑虑，担心前端页面结构（如层级深度）变化会导致绝对路径失效，同时需要实现在 XPath 中传入变量进行动态定位。
*   **大型流程架构设计缺乏最佳实践**：面对如“巡检任务”等包含多个用例的复杂业务，用户不确定应将所有逻辑集中于单一应用中，还是按用例拆分为多个独立应用，缺乏合理的工程架构指导。

## 4. 解决方案 (Resolution)
Astron 技术支持团队通过远程协助和详细指导，提供了以下解决方案：
1.  **离线镜像导入指导**：
    *   指导用户检查 Docker 镜像的导出与导入过程，确保镜像标签（Tag）和完整性正确。用户经重新导出操作后，成功在离线环境加载了镜像。
2.  **交互逻辑优化**：
    *   **启用模拟人工输入**：针对登录按钮无法激活的问题，指导用户在 RPA 输入组件的参数设置中勾选**“模拟人工输入”**。该功能模拟了真实的键盘敲击事件，成功触发了页面的 JS 校验逻辑，使登录按钮变更为可点击状态。
    *   **处理安全警告**：建议通过 CV（计算机视觉）点击或增加异常处理逻辑（点击“高级”->“继续前往”）来自动绕过 HTTPS 安全警告页。
3.  **功能操作与排查指引**：
    *   **智能录制教学**：通过远程演示，向用户明确了智能录制生成流程的存放位置（侧边栏菜单），并强调录制功能仅为辅助，复杂场景仍需人工微调。
    *   **日志定位**：提供了客户端日志的标准路径 `%APPDATA%/astron-rpa/logs`，帮助用户在遇到启动失败等问题时自主收集排查信息。
4.  **流程开发与组件化指导**：
    *   **变量与返回值操作指导**：通过截图演示了“设置变量”组件的使用方法，并指导用户在 Python 模式下提取脚本返回值（如 `script_result["item_list"]`）。同时解答了组件间元素作用域隔离及元素重名互不影响的机制。
    *   **XPath 动态定位优化**：向用户解释了 RPA 捕获元素的机制（默认锁定 ID 及层级路径）。指导用户在遇到页面层级变动或动态元素时，可通过取消勾选动态属性（如变化的 ID）、提取相对路径，或直接在 XPath 表达式中嵌入变量来实现灵活且稳定的定位。
    *   **大型项目架构建议**：针对复杂流程的组织方式，为用户提供了相关案例和官方资源参考，引导用户根据实际业务用例进行合理的项目拆分与架构设计。
5.  **AI模型接入与系统集成支持**：
    *   **AI接入指引**：引导用户通过“扩展组件 -> AI -> 自定义模型”接入大模型单轮对话原子能力，满足了初步的自定义 AI 交互需求。
    *   **Open API 集成**：针对用户通过 Jenkins 调度 RPA 任务的需求，提供了 Open API 官方文档（`https://www.iflyrpa.com/docs/open-api/overview.html`），帮助打通外部调度系统。

## 5. 客户价值 (Value)
*   **打通离线落地路径**：协助客户成功跑通了从互联网环境获取资源到内网离线环境部署的全流程，验证了 Astron RPA 在高安全隔离环境下的部署可行性。
*   **解决自动化核心卡点**：解决了模拟输入、环境差异带来的交互难题，确保了 POC（概念验证）阶段核心流程的顺利执行，增强了客户对产品的信心。
*   **赋能复杂业务扩展**：通过 API 调度支持与自定义大模型接入指引，为客户未来将 RPA 融入整体 CI/CD 流程及智能化办公场景奠定了基础。

## 6. 产品共创与反馈 (Co-creation & Feedback)
基于项目实战，客户提出了以下反馈与建议，为产品优化提供了方向：
*   **平台架构优化（已规划至 4 月新版本）**：
    *   **团队协作修复**：修复 Casdoor 多数据源导致的团队邀请查询 Bug，恢复正常的多用户协作体验。
    *   **内网 AI 全面支持**：升级服务端源码，使智能组件能够无缝对接客户内网部署的自定义大模型。
    *   **参数配置增强**：支持组件参数的“必选/可选”配置，提升组件化开发的严谨性。
*   **底层组件容错性提升**：反馈了“关闭浏览器”组件底层 DLL 报错的必现问题，推动研发团队进一步排查底层组件的跨环境稳定性。
*   **性能优化**：建议针对涉密终端等特定硬件环境进行客户端性能优化，减少资源占用和卡顿。
*   **智能录制体验提升**：建议优化智能录制的交互设计，在录制完成后给予更明确的反馈（如自动跳转或提示生成结果位置），并提升对动态/复杂元素的自动识别能力。
*   **跨语言环境适配**：用户关注英文环境下开发的流程在中文环境的兼容性，提示产品需增强选择器的鲁棒性，减少对界面文本语言的依赖。
*   **变量类型校验优化**：用户反馈在判断整数变量（如 `index_2`）是否为 0 时仍需强制类型转换，建议优化底层的隐式类型转换与校验判断逻辑。
*   **多用户权限与应用共享**：在多用户协作场景下，用户希望平台能提供更清晰的应用共享机制，使团队成员之间能方便地查看和调用彼此发布的应用。
*   **调试信息增强**：建议在调试窗口中提供更详尽的逻辑判断结果（如明确显示“元素是否存在”的 True/False 状态），以便开发者快速定位逻辑错误。
*   **组件化开发体验优化**：建议支持对组件输入参数的顺序进行调整（如拖拽排序），以更符合开发者的使用习惯；同时优化组件传参的交互引导，降低新用户的学习门槛。
