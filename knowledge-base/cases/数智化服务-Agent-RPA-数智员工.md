# 用户案例：某数智化服务企业 Astron 数智员工

*信息来源：项目沟通记录及用户反馈 (2026-01-12 ~ 2026-03-18)*

---

## 1. 案例概况 (Case Overview)

*   **项目名称**：某数智化服务企业数智员工智能体应用
*   **客户方**：某数智化服务企业
*   **应用产品**：Astron Agent, Astron RPA
*   **部署状态**：开发环境（联调中）
*   **核心场景**：数智员工智能体应用

## 2. 业务背景 (Background)
某数智化服务企业正在利用 Astron Agent 和 Astron RPA 构建“数智员工”智能体应用，旨在通过自动化和智能化技术提升业务处理能力。目前项目处于开发与联调阶段，重点在于验证 Astron 平台与企业现有后端业务系统的集成能力，特别是通过插件（Plugin）机制实现数据交互与业务流程编排。

## 3. 挑战与痛点 (Challenges)
在智能体工作流的开发与联调过程中，技术团队遇到了以下阻碍与需求：

1.  **后端接口兼容性问题**：
    *   **插件入参格式限制**：Astron Agent 的 HTTP 插件目前默认支持 JSON Object（键值对）格式的请求体。
    *   **API 接口不兼容**：客户现有的后端业务接口要求请求体必须为顶层 **JSON Array** 格式（例如 `[{"skuId":...}, {"skuId":...}]`）。
    *   **集成报错**：直接封装列表会导致后端反序列化失败，且修改后端接口成本过高。

2.  **复杂任务编排能力不足**：
    *   **任务拆分需求**：在构建复杂业务流时，客户发现当前的 Core 层缺乏对“复杂任务拆分节点”的原生支持，难以实现由 AI 自动规划并调用 Agent Skill 的高级场景。
    *   **Skill 功能支持**：希望增强对 Agent Skill 的集成与支持，以满足更灵活的业务逻辑。

3.  **RPA 部署与配置问题**：
    *   **环境部署受阻**：在私有化部署 Astron RPA 开源版时，遇到 Docker 容器（如 MySQL）启动失败及端口冲突问题，导致服务无法正常运行。
    *   **版本差异咨询**：对开源版与商业版的功能差异（如 MCP 节点支持、自定义配置等）存在疑问，并确认了 RPA 工具需依赖后端服务部署。

4.  **工作流执行与管理问题**：
    *   **死循环异常**：用户反馈在部署后运行简单工作流时，调用 `workflow/chat` 出现死循环输出的情况。
    *   **智能体复制报错**：用户发现在非个人空间下复制智能体时会触发报错。

5.  **前端显示与交互异常**（新增）：
    *   **输入框类型错误**：发布到广场后，字符串类型的输入框错误显示为文件上传框（已确认前端 Bug）。
    *   **输出参数丢失**：在工作流中设置 Object 类型变量时，偶发下一节点无法获取输出参数的情况。
    *   **模版创建智能体数据缺失**：通过模版创建智能体时，由于前端表单未传递模版信息，导致后端 `chat_bot_base` 表的 `bot_template` 字段为空，该 Bug 在新版本中依然存在。
    *   **工作流导入数据丢失**（2026-03-28新增）：用户反馈导入工作流后，`chat_bot_list` 表无数据，导致在智能体列表中无法看到导入的智能体（Issue #1119），严重影响产品体验稳定性。

6.  **扩展能力与运维限制**（新增）：
    *   **代码节点依赖**：用户咨询如何在 Python 代码节点中安装第三方依赖库（目前暂不支持动态安装）。
    *   **多模态模型支持**：开源版图像/视频生成功能依赖讯飞付费资源包，不支持直接使用硅基流动等兼容接口。
    *   **AITools 服务异常**：私有化部署中遇到 MinIO/OSS 连接 403 Forbidden 报错，涉及域名转发配置问题。

## 4. 解决方案 (Resolution)
针对上述问题，Astron 技术支持团队与客户进行了深入协作：

1.  **接口兼容性临时方案**：
    *   建议在工作流中改用 **代码节点 (Code Node)** 替代标准插件节点，通过编写自定义脚本（Python/Node.js）直接构建 JSON Array 请求体，成功绕过插件配置限制，保障了联调进度。

2.  **产品功能共建 (Feature Request & Bug Report)**：
    *   **Issue 提交**：客户积极参与开源共建，已提交多个 Issue：
        *   `[FEATURE] 对于Agent Skill的功能支持及任务拆分节点的支持 #878`
        *   `[BUG] 代码仓库release分支目录下，没有迁出版本分支 #925`（官方已回应 v1.0.1 分支可用）
        *   `[BUG] 选择非个人空间，复制智能体报错 #941`
        *   `[BUG] 通过模版创建智能体，bot_template字段无值`（用户通过手动插入 `maas_template` 表临时规避，官方已跟进排查）
        *   `[BUG] 导入工作流在chat_bot_list表没数据 #1119`（官方已跟进排查修复）
    *   **路线图对齐**：Astron 团队确认了需求，并透露后续版本计划将“插件/工具”全面向 "Skills" 升级。

3.  **商业化与部署咨询**：
    *   **功能与报价**：Astron 团队提供了企业版功能清单与报价方案。
    *   **模型与成本**：明确了商业版报价包含底层的**私有化星火模型**。
        *   **Token 费用**：私有化部署调用本地模型**无 Token 费用**；若配置调用其他云端大模型（如 GPT-4 等）则需额外付费。

4.  **技术答疑与最佳实践**（新增）：
    *   **客户端输出规范**：确认智能体输出多模态内容（文字、图片、代码）时，前端解析标准为 **Markdown** 格式。
    *   **模型适配建议**：用户反馈在使用 `deepseek-reasoner` 时因思考过程过长导致超时或显示异常，切换为 `deepseek-chat` 后恢复正常；Astron 侧提示可关注 `CODE_EXEC_TIMEOUT_SEC` 配置，但核心侧超时机制较难直接修改。

## 5. 客户价值 (Value)
*   **保障联调进度**：通过代码节点替代方案，解决了核心业务接口的联调阻塞问题。
*   **产品共建与社区贡献**：客户不仅反馈了通用性需求，还主动提交 Feature Request 及 Bug Issue (#878, #925, #941) 参与开源共建。Astron 计划在官网及仓库展示墙收录客户 Logo，以表彰其作为早期种子用户的贡献。
*   **明确技术演进方向**：通过深入沟通，客户明确了 Astron 平台在 Agent Skill 和客户端形态上的演进规划，为双方后续的深入合作与商业化探索奠定了基础。
