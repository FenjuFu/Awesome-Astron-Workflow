# 用户案例：某企业软件服务商 Astron RPA 构建环境搭建

*信息来源：项目沟通记录及用户反馈 (2026-02-04 ~ 2026-02-06)*
---

## 1. 案例概况 (Case Overview)
*   **项目名称**：某企业软件服务商企业级 ERP 自动化项目
*   **客户方**：某企业软件服务商
*   **应用产品**：Astron RPA / Astron Agent (开发环境构建)
*   **部署状态**：开发与环境搭建阶段
*   **核心场景**：企业级 ERP 场景自动化

## 2. 业务背景 (Background)
某企业软件服务商致力于将 Astron RPA 技术应用于企业级 ERP 场景，以提升业务流程自动化水平。在项目启动初期，技术团队需要基于 Python 3.13 环境构建 Astron 运行环境，以便进行后续的二次开发与业务流程定制。

## 3. 挑战与痛点 (Challenges)
在环境搭建与构建（Build）过程中，技术团队遇到了以下关键技术瓶颈，导致 `build.bat` 脚本执行失败，阻碍了开发工作的开展：
*   **C++ 编译环境缺失**：在 Windows 环境下构建依赖包时，由于缺少 Microsoft Visual C++ 14.0 或更高版本，导致部分需要编译的 Python 库（如 `pywinhook`）安装失败。
*   **权限与环境隔离问题**：即使安装了编译工具，在普通用户权限下仍出现 `pywinhook` 和 `psutil` 构建错误，提示权限不足或环境调用受限。
*   **数据库驱动构建失败**：
    *   **`psycopg2`**：PostgreSQL 数据库适配器在构建时报错，即使尝试替换为 `psycopg2-binary` 也未能完全解决环境兼容性问题。
    *   **`cx-oracle`**：Oracle 数据库驱动在安装时同样出现构建错误，阻断了依赖安装流程。
*   **依赖管理复杂性**：项目 `pyproject.toml` 中包含了当前阶段尚未使用的数据库组件依赖，导致构建过程因为非核心组件的安装失败而整体中断。

## 4. 解决方案 (Resolution)
Astron 技术支持团队通过实时协作，提供了针对性的环境修复方案：
1.  **环境补全指导**：明确指出报错原因，指导用户安装 Microsoft Visual C++ 14.0+ 构建工具（需包含 MSVC v143 和 Windows 10/11 SDK），解决了基础编译环境缺失的问题。
2.  **权限调整**：针对 `pywinhook` 等库的持续报错，建议用户切换至 **Administrator** 管理员账户进行构建，成功解决了因权限不足导致的编译挂起和失败问题。
3.  **依赖剪裁与优化**：
    *   针对构建报错的 `psycopg2` 和 `cx-oracle` 库，经确认当前 ERP 场景暂时无需使用这两个数据库组件。
    *   指导用户修改 `/engine/components/astronverse-database/pyproject.toml` 文件，移除相关依赖条目。
    *   通过剪裁非必要依赖，成功绕过了复杂的本地编译过程，确保了核心环境的顺利构建。

## 5. 客户价值 (Value)
*   **快速扫清开发障碍**：协助客户快速定位并解决了环境搭建中的“拦路虎”，将环境准备时间从可能的多天缩短至数小时内。
*   **降低上手门槛**：通过合理的依赖剪裁方案，避免了在开发初期陷入复杂的底层驱动编译问题，使团队能更专注于 ERP 业务逻辑的实现。
*   **保障项目启动**：确保了开发环境的成功运行，为后续 ERP 自动化流程的开发与部署奠定了基础。

## 6. 产品共创与反馈 (Co-creation & Feedback)
此次支持过程也为 Astron 产品优化提供了参考：
*   **依赖按需加载**：建议后续版本优化 `pyproject.toml` 依赖管理，将非核心数据库驱动（如 Oracle、PostgreSQL）设为可选依赖（Optional Dependencies），避免因个别组件构建失败影响整体环境安装。
*   **环境预检工具**：建议在构建脚本中增加环境预检功能，提前检查 C++ 构建工具等必要前置条件并给出明确提示，提升开发者体验。
